# 00 · 前言：为什么 agent 逻辑需要单独讲

## 一个观察

如果你把 gstack 的 55 个 SKILL.md 铺开，会发现里面 **不是代码**。它们是给 LLM 看的指令。这些指令按一套隐性协议组织：什么时候该问用户、什么时候该停下、遇到哪种输入走哪个分支、把哪种状态写回磁盘。

这套协议就是 gstack 的 agent 逻辑。它是这个仓库真正的 IP。

**基础设施是可替代的**：不用 Bun 用 Node 也可以，不用 Chromium 用 Firefox 也可以，不用 SKILL.md 用 JSON 也可以。**但 agent 逻辑不可替代** —— 换掉它，gstack 就变成另一个 skill 集合，不再是它自己。

## 与上一版的区别

上一版 gstack-book 覆盖 6 大部分（Foundation / Core Subsystems / Workflows / Platform Extensions / Engineering Practice / Capstone），15 章一半在讲基础设施（模板引擎、browse daemon、多 host 分发、Bun 构建、测试分级）。**用户读完知道 gstack "是什么"，但不知道 LLM 在 gstack 里"做什么决策"**。

这一版把范围收紧到只讲 agent 逻辑：

| 上一版 | 这一版 |
|---|---|
| 讲 SKILL.md.tmpl → SKILL.md 怎么生成 | 只讲生成后的 SKILL.md 里 LLM 读到什么 |
| 讲 browse daemon 的 Bun.serve + CDP | 只讲 skill 里 `$B` 命令背后的 agent 决策语义 |
| 讲 multi-host 的 HostConfig.pathRewrites | 只讲 `boundaryInstruction` 这种"跨 host prompt injection 防御" |
| 讲 free / paid evals 分层 | 不讲测试 |
| 讲 VERSION / CHANGELOG 双写 | 不讲发布 |

## 一个心智模型：把 skill 当函数看

对 LLM 来说，一个 skill 是一个"带输入、带输出、带副作用"的函数：

```text
skill(user_prompt, preamble_state) →
    { AskUserQuestion?, Bash?, Edit?, ... } × N steps
    → STATUS: DONE | DONE_WITH_CONCERNS | BLOCKED | NEEDS_CONTEXT
```

- **输入** = 用户 prompt + preamble bash 输出的 `KEY: value` 状态
- **输出** = 一系列 tool call
- **副作用** = 落盘的 learnings / plan file / decisions
- **返回值** = STATUS token

本书按这个函数模型的 5 个部位组织：

| Part | 部位 | 关键问题 |
|---|---|---|
| 1 · 输入层 | 输入 | LLM 到底拿到多少状态？tier 决定的密度是什么？|
| 2 · Router 与编排 | 分派 | 一个模糊请求怎么变成确切 skill 调用？skill 之间怎么串？|
| 3 · Plan-mode Agent | 前置决策 | 动代码前，agent 怎么把方案打磨到位？|
| 4 · Execution Agent | 执行 | 真动代码时，agent 被哪些硬约束绑住？|
| 5 · 记忆与安全 | 副作用与边界 | agent 学到的东西存哪、怎么防止越权 |

Capstone 反过来：给你一份"只包含 agent 逻辑" 的 skill 模板，让你能自己写。

## 一个承诺

本书只讲 agent 逻辑意味着 **每一段引用都指向"LLM 会读到 / 遵守 / 决策的东西"**。基础设施代码（Bun.serve / CDP / _link_or_copy）如果出现，只是为了标出"这里是 agent 边界之外" 而不再展开。

那么，开始吧。
